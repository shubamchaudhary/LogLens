## Locked design decisions (with their defense in 02)

1. **Two-layer extraction** ‚Äî parsers for numbers, LLMs for meaning; enrich anomalous windows only (02 ¬ß5).
2. **Kafka partition-per-API-key** for LLM lanes; at-least-once + idempotent handlers (fingerprint dedup, natural-key upserts); retry topic for 429s; DLQ. Four topics total (02 ¬ß7).
3. **Table-per-session for raw chunks** (`log_chunks_s_{sessionId}`, own HNSW **+ GIN full-text**) ‚Äî justified by corpus-scale sessions (10‚Åµ‚Äì10‚Å∂ chunks); ceiling ~tens-of-thousands of tables; exit = hash partitioning; names from validated UUIDs only (02 ¬ß6).
4. **Query time = SQL over materialized tables**; drill-down RAG (Graph 2, session-scoped pgvector + GIN) is the only query-time LLM path (02 ¬ß9).
5. **CUT LIST (refinement pass, do not re-add without cause):** Elasticsearch, RRF/hybrid fusion, durable object storage (see #8), query-time recursive summarization (legacy doc-QA mode only), async query API + status topic (SSE reads the DB status row), dedicated vector DB, Kafka exactly-once. Reasons in 02 ¬ß4 ‚Äî that section is interview material.
6. **Chunk logs by time window**, not by page/size.
7. **$0 constraint**: whole stack free/self-hosted; Gemini free tier.
8. **Staging blob lifecycle (owner-specified):** uploads land in **MinIO** (self-hosted, S3-compatible, free ‚Äî one docker-compose service, used from local dev through prod, no separate "local disk" code path); chunker *streams* the file (never whole-in-memory); after processing the staged file is **deleted** (`documents.staged_file_deleted`) ‚Äî the chunks in Postgres are the durable copy (02 ¬ß6).
9. **Core entity tables (owner-specified):** `users` (user_id PK, email, password_hash, full_name, created_at, last_login_at, is_active ‚Äî matches v1 init.sql), `sessions` (session_id PK, user_id FK, status+counters), `documents` (document_id PK, user_id FK, session_id FK, file_url, status). ER diagram in 02 ¬ß6.

## Facts that save you an hour of re-reading

- Backend = ONE Gradle module `deepdocai-backend`, packages `api/common/core/data/llm`.
- Key concurrency today is **thread-pinned key slots** (NOT round-robin) ‚Äî 5 keys, 7.5s/key, 429 ‚Üí re-enqueue + 60s sleep. Maps 1:1 to Kafka partition-per-key; that's the central migration insight.
- Job queue was `processing_jobs` + 3s polling + lock leases (`ProcessingJobWorker`) ‚Äî **DELETED in Phase 1** (lives in git history only).
- DB on port **5434** via docker-compose; schema in `init.sql` (no Flyway yet).
- Transactions deliberately split around long LLM calls (short TX ‚Üí no TX ‚Üí short TX) ‚Äî preserve in refactors.
- Existing prompts already log-analysis flavored (`DocumentAnalysisPrompts` mentions stack traces, thread states, pool events) ‚Äî the pivot matches the code's direction.
- Vector search today: `ORDER BY embedding <=> :q LIMIT 500` two-query pattern in `DocumentChunkRepositoryImpl`; known issues: could be single query; no `ef_search` tuning; `LIMIT 500` > default `ef_search=40` is incoherent. Superseded by target design's per-session tables.
- Frontend `examprep-frontend` still has old naming; low priority. Git branch `main`; write real commit messages (history is full of "m").
